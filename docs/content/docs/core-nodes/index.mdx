Core nodes ship with the **core** plugin (`plugins/core/index.ts`). They cover triggers, generic HTTP and AI actions, branching, PostgreSQL reads, sub-workflows, and small data-shaping steps. Each page lists the node **kind** string (stored on the graph), the **family** (library section), Inspector fields, and how the runtime behaves when that step runs.

## Triggers

| Page | Kind | Summary |
| --- | --- | --- |
| [Button press](./button) | `button` | Manual runs from the app |
| [Schedule](./schedule) | `schedule` | Cron-based dispatch |
| [Receive webhook](./webhook) | `webhook` | HTTP ingress with optional path suffix and shared secret |
| [Context from parent workflow](./parent-context) | `parentContext` | Sub-workflow entry only |

## Source layout

Runtime handlers live under `plugins/core/steps/<step-id>.ts` and map from each node’s `stepId` in the manifest. Triggers do not use step files; execution records them as synthetic completed steps with the trigger payload.

For adding or changing nodes, see [Plugin development](/docs/plugin-development).
