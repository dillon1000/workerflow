Plugins are now folder-based and auto-discovered at build time. If you add a new plugin under `plugins/<plugin-id>/`, it can:

- add nodes to the Node library automatically
- render typed fields in the Inspector automatically
- validate config in both the UI and on publish
- provide runtime step handlers on the Cloudflare Worker
- register trigger matching and signature verification
- register connection test logic

You should not need to modify system files like `apps/web/worker/services/runtime.ts`, `apps/web/worker/routes/triggers.ts`, or `apps/web/src/lib/workflow/templates.ts` to add a normal plugin.

## Quick Start

To scaffold a new plugin:

```bash
pnpm create-plugin
```

To refresh the generated plugin index after adding, removing, or renaming plugin folders:

```bash
pnpm discover-plugins
```

The scaffold creates a starter plugin from `plugins/_template/`. The discovery script regenerates `plugins/index.ts`, which tracks the plugin folders available in the repo.

## Folder Structure

Each plugin should follow this shape:

```text
plugins/<plugin-id>/
  index.ts                # plugin manifest
  credentials.ts          # optional shared credential types
  icon.tsx                # optional icon for editor cards
  steps/
    <step-id>.ts          # runtime step handlers
  test.ts                 # optional connection test
  trigger.ts              # optional trigger matcher/verifier
```

## What Is Auto-Discovered

The app and worker discover plugins with `import.meta.glob` from these locations:

- `plugins/*/index.ts`
- `plugins/*/icon.tsx`
- `plugins/*/steps/*.ts`
- `plugins/*/test.ts`
- `plugins/*/trigger.ts`

The repo also maintains a generated index at `plugins/index.ts` via `pnpm discover-plugins`. That file is for tooling and should not be edited manually.

That means:

- adding a node to `index.ts` makes it show up in the Node panel
- adding `fields` to the node manifest makes them show up in the Inspector
- adding `stepId` plus a matching `steps/<step-id>.ts` file makes it executable
- adding `test.ts` makes provider connection testing available
- adding `trigger.ts` makes inbound trigger routing available at `/api/triggers/:kind`

## Manifest Contract

Each plugin exports:

```ts
export const plugin: WorkflowPluginManifest = {
  id: "github",
  title: "GitHub",
  description: "Issues and pull request automations.",
  connections: [...],
  nodes: [...],
};
```

## Node Definitions

Each node definition supports:

- `kind`: globally unique node kind
- `family`: `trigger`, `action`, `logic`, or `data`
- `title`
- `subtitle`
- `accent`
- `defaultConfig`
- `stepId`: required for executable non-trigger nodes
- `executionMode`: optional, either `step` or `inline`
- `fields`: Inspector field definitions and validation rules

Example:

```ts
{
  kind: "githubAction",
  family: "action",
  title: "GitHub create issue",
  subtitle: "Create a GitHub issue with real API calls.",
  accent: "from-stone-900 via-stone-800 to-stone-700",
  defaultConfig: {
    connectionAlias: "",
    owner: "",
    repo: "",
    title: "Workflow-generated issue",
    body: "Created from workflow output: {{ trigger.data }}",
    labels: "automation",
    assignees: "",
  },
  stepId: "create-issue",
  fields: [
    {
      key: "connectionAlias",
      label: "Connection",
      kind: "connection",
      required: true,
      connectionProvider: "github",
    },
    {
      key: "owner",
      label: "Owner",
      kind: "text",
      required: true,
    },
    {
      key: "title",
      label: "Issue title",
      kind: "text",
      required: true,
      allowTemplates: true,
    },
  ],
}
```

## Inspector Fields

Supported field kinds:

- `text`
- `textarea`
- `number`
- `json`
- `select`
- `connection`
- `workflow`

Supported field options:

- `required`
- `placeholder`
- `description`
- `allowTemplates`
- `min`
- `max`
- `options`
- `connectionProvider`

The Inspector uses these definitions to render the correct control and show inline validation errors. Validation also runs server-side during graph validation, so publish checks match the editor.

## Runtime Step Handlers

Step handlers live in `steps/<step-id>.ts` and export:

```ts
export const run: WorkflowStepRunner = async (context) => {
  return {
    detail: "Created issue #123.",
    output: { id: 123, url: "..." },
  };
};
```

Use the provided context helpers instead of reimplementing plumbing:

- `render(value)` for `{{ trigger.data }}` style templating
- `parseList(value)`
- `parseMaybeJson(value)`
- `evaluateExpression(expression)`
- `getConnection(alias)`
- `getConnectionSecret(connection, keyName)`

### Execution Modes

Most steps should use the default `executionMode: "step"`, which wraps the handler in `step.do(...)`.

Use `executionMode: "inline"` only when the handler must call workflow primitives directly, such as `step.sleep(...)`. The built-in `wait` plugin uses this mode.

## Trigger Plugins

Trigger plugins can export:

```ts
export const triggerHandlers: WorkflowTriggerHandler[] = [
  {
    kind: "github",
    preparePayload({ payload, headers }) {
      return {
        ...payload,
        githubEvent: headers.get("x-github-event") ?? "",
      };
    },
    matches(node, payload) {
      return true;
    },
    async verify(context) {
      // signature verification
    },
  },
];
```

This registers the trigger automatically for `POST /api/triggers/:kind`.

## Connection Schemas

If your plugin needs reusable credentials, add a `connections` entry in the manifest. That makes the provider show up in the Connections UI automatically.

Recommended rules:

- use `fetch` directly for provider APIs
- do not add SDK dependencies
- keep secrets in app-managed reusable connections
- do not add env fallbacks for provider webhook secrets

## Validation Behavior

The graph validator now checks:

- exactly one trigger node exists
- all node kinds are registered
- all configured fields pass plugin validation
- condition nodes have both `true` and `false` branches
- non-trigger nodes have an incoming edge
- webhook `pathSuffix` values are not duplicated
- graph edges reference real nodes
- cycles are not present

## Cloudflare Runtime Notes

The Cloudflare worker no longer has provider-specific action cases in `runtime.ts`. Instead:

- node kinds resolve through the discovered plugin registry
- step handlers are loaded from `plugins/*/steps/*.ts`
- trigger verification is loaded from `plugins/*/trigger.ts`
- connection testing is loaded from `plugins/*/test.ts`

The remaining worker orchestration code is generic workflow execution plumbing, not provider-specific action registration.

## Checklist

Before finishing a plugin:

1. Run `pnpm create-plugin` or add the plugin folder and manifest manually.
2. Run `pnpm discover-plugins` after adding, removing, or renaming a plugin folder.
3. Add any required `steps/`, `trigger.ts`, and `test.ts`.
4. Confirm the node appears in the Node panel.
5. Confirm the Inspector fields render and validate.
6. Confirm publish succeeds with valid config and fails with invalid config.
7. Run `pnpm type-check`.
8. Run `pnpm fix`.

If your plugin talks to a third-party API, also verify at least one real request path end-to-end.
